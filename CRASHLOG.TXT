Common factors:
  - All crashes occur right after the LLM returns (stopReason=toolUse)
  - Cache read values are very high (67K–235K tokens) — the conversation
    context is growing large with each tool call result
  - Crashes often reported 2-3 times in quick succession for the same
    event (likely the user retrying / pi.dev re-rendering)
  - No system-level segfaults or OOM kills in dmesg

Likely cause:
  - The conversation context is ballooning.  Each file read (e.g. dwm.c
    at 65KB) gets cached and re-sent on every subsequent request.
    After many reads, the context window fills up and something in the
    pi.dev TUI or LLM pipeline breaks.
  - The deepseek-v4-pro model may have tighter context limits than
    expected, or pi.dev's context management may not be trimming
    aggressively enough.

Mitigation ideas:
  - Use raw reads (raw: true) for large files — saves LINE#HASH overhead
  - Periodically start fresh sessions to reset context
  - Avoid reading large files repeatedly; use grep to target specific
    sections
